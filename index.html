Project Scenario

Background & the mid-project pivot challenge

🎯 The Setting

We are the junior project managers of a university startup developing Smart-Attend — a mobile app that uses Geo-fencing technology to automatically mark student attendance when they physically enter a classroom zone (within 10 metres). The app includes:

  • Student login via university ID
  • Teacher dashboard with real-time attendance list
  • GPS-based geo-fencing for automatic attendance marking
  • Push notifications for students upon attendance marking
  • Admin override for technical failures

⚡ The Pivot — Biometric Requirement

Halfway through development (month 3 of a 6-month plan), the Dean introduces a new mandatory requirement: Biometric Face ID Verification must be integrated into the app to prevent "proxy attendance" — where a student stands near the classroom but doesn't actually attend. This requirement fundamentally changes the system architecture.

Task 1 — The "Waterfall" Failure Analysis

Why the traditional Waterfall model collapses under mid-project requirements change

Reason 1

Rigidity — The No-Return Phases

The Waterfall Model is built on a strictly sequential, phase-gated structure: Requirements → Design → Development → Testing → Deployment. Each phase is signed off as "complete" before the next begins, and the model explicitly prevents revisiting earlier stages. When the Dean announces the Biometric requirement in Month 3, development of the geo-fencing module is already underway. Under Waterfall, the team cannot legally go back to the Requirements or Design phases without invalidating months of approved work.

🔴 Smart-Attend Impact: The original system design never accounted for a camera permission, a Face ID API integration, or a liveness-detection algorithm. Whole architecture documents and design diagrams from Phase 1 & 2 become obsolete overnight, yet the model offers no mechanism to gracefully re-enter those phases. The project effectively breaks its own process.
Reason 2

Cost of Change — The "Frozen Blueprint" Penalty

In Waterfall, the detailed design document acts as a frozen blueprint. Every line of code written, every module tested, and every database schema created is built on that blueprint. Introducing Biometric Verification mid-project isn't just an "add-on" — it requires redesigning the user authentication flow, adding a new Face ID module, updating privacy policies, expanding the database to store biometric consent logs, and re-testing the entire login and attendance pipeline. In systems engineering, the cost of a requirement change in the Development phase is estimated to be 10–100× more expensive than catching it in Requirements.

🔴 Smart-Attend Impact: Weeks of already-written login logic (US01), notification systems (US04), and geo-fencing code (US03) must be ripped out and rewired to embed face verification as a gate. The budget overruns and schedule delays (estimated 6–8 additional weeks) turn the 6-month project into a 9-month project — likely past the academic year deadline.
Reason 3

Delayed Testing — The "Big Bang" Integration Failure

Waterfall defers all integration and system testing to Phase 4 — after all modules are developed. This is the "Big Bang" integration approach: everything is assembled at once and tested collectively at the very end of the development cycle. Adding Biometric Face ID after geo-fencing is already developed means both subsystems must integrate for the first time during this final testing phase. Network latency between GPS detection and Face ID camera activation, camera API conflicts on different Android/iOS devices, biometric data handling edge cases — none of these are discovered until Week 23 of a 24-week project, leaving no viable time for fixes.

🔴 Smart-Attend Impact: In a pilot scenario: Geo-fencing correctly detects a student within 10m, but the Face ID camera takes 4–7 seconds to initialize, causing the geo-fence window to expire before verification completes — attendance is not marked. This bug, discovered in Week 23, cannot be meaningfully fixed without restructuring both modules. Under Waterfall, the project ships broken or not at all.

Waterfall vs Agile — At a Glance

❌ Waterfall (Traditional)

  • Sequential, phase-locked progression
  • Requirements frozen at project start
  • Delivery only at end (Month 6)
  • Customer sees product at the very end
  • Changes are expensive and disruptive
  • Testing is deferred to final phase

✅ Agile / Scrum (Adaptive)

  • Iterative, incremental development
  • Requirements evolve with each Sprint
  • Working product delivered every 2 weeks
  • Continuous stakeholder feedback loops
  • Changes absorbed at low cost via backlog
  • Testing integrated within every Sprint

Task 2 — The "Agile" Pivot (Scrum Framework)

Restructuring Smart-Attend development into two 2-week Sprints

Sprint 1

Minimum Viable Product (MVP)

📅 Weeks 1–2  |  2-week iteration
Sprint Goal
Deliver a working, student-usable app with login, real-time attendance list, and geo-fencing — so early user testing can begin immediately.
Deliverables
  • Basic UI screens: Login, Home, Attendance Status
  • User Login via University ID (US01)
  • Core Geo-fencing logic — 10m detection (US03)
  • Teacher real-time attendance dashboard (US02)
  • Attendance push notification to student (US04)
  • Admin manual override panel (US05)
Sprint 2

The Biometric Update

📅 Weeks 3–4  |  2-week iteration
Sprint Goal
Integrate Face ID Biometric Verification as an anti-proxy gate and refine the reporting dashboard based on Sprint 1 user feedback.
Deliverables
  • Face ID / Biometric API integration
  • Liveness detection to prevent photo spoofing
  • Biometric consent & privacy policy screen
  • Biometric data encryption & secure storage
  • Enhanced analytics & reporting dashboard
  • Bug fixes from Sprint 1 retrospective

🔄 Scrum Ceremonies — How the Team Operates Each Sprint

📋

Sprint Planning

Team selects tasks from Product Backlog; defines Sprint Goal and estimates hours.

☀️

Daily Scrum

15-min standup: What did I do? What will I do? Any blockers?

🚀

Sprint Review

Team demos the working product to the Dean & stakeholders for feedback.

🔍

Retrospective

Internal review: What went well? What to improve in next Sprint?

🔀 How Scrum Handled the Biometric Pivot

When the Dean added Face ID mid-project, the Scrum Master simply added new user stories to the Product Backlog. During Sprint 2 Planning, the team reprioritized these items to the top. No phase was "violated." No existing Sprint 1 code was discarded — it became the foundation Sprint 2 builds on.

📊 The Cost of Change — Agile vs Waterfall

In Waterfall, the biometric change would cost an estimated 6–8 additional weeks and significant rework. In Scrum, the change was absorbed into Sprint 2 backlog refinement during a 1-hour planning session. The working MVP from Sprint 1 was unaffected and ready to demo to users throughout.

📦 Definition of Done (DoD)

A user story is considered "Done" when: code is written & peer-reviewed, unit tests pass, feature is integrated into the main branch (via CI pipeline), and the Product Owner has accepted the demo during Sprint Review.

🎓 Conclusion — Why Agile Wins for Smart-Attend

The Biometric requirement pivot is a real-world scenario that perfectly illustrates why Waterfall fails and Agile thrives in modern software development. Waterfall's rigid phase structure, expensive change mechanism, and deferred testing would have turned the Dean's single requirement into a project-ending crisis. Scrum, by contrast, absorbed the change gracefully: Sprint 1 delivered a working MVP to real users within 2 weeks, generating valuable feedback, while Sprint 2 incorporated the biometric requirement as a natural extension. The total delivery timeline remained on schedule, and the team maintained stakeholder confidence throughout — the hallmark of a truly Agile organisation.
